# PowerFS 设计原则

## 网络协议原则

### 核心原则：元数据和数据通信必须使用 gRPC（HTTP/2）

- **gRPC** 是分布式文件系统的首选协议，基于 HTTP/2，具有以下优势：
  - 多路复用：单一连接上同时处理多个请求/响应
  - 头部压缩：HPACK 压缩减少网络开销
  - 双向流：支持心跳、实时同步等场景
  - 强类型：Protobuf 定义确保类型安全
  - 高性能：二进制协议，处理效率高

- **tonic gRPC 库的 URL 写法**：
  - tonic 使用 `http://` 前缀来标识 gRPC 服务地址
  - 底层自动协商并升级到 HTTP/2
  - `.http2_keep_alive_interval()` 等配置证明使用的是 HTTP/2

- **允许使用 HTTP 的场景**：
  - 管理界面（如 Web UI）
  - REST API（可选，为外部系统提供接口）
  - 监控和健康检查（简单场景）

## 测试原则

### 性能测试必须使用标准工具

- **fio**：标准的 I/O 性能测试工具
  - 支持多种 I/O 引擎（sync, libaio, posixaio 等）
  - 支持多种测试模式（顺序读写、随机读写、混合负载）
  - 输出标准指标（IOPS、带宽、延迟等）
  - 可配置队列深度、线程数、文件大小等参数

- **禁止自定义性能测试程序**：
  - 自定义程序无法模拟真实负载
  - 无法准确测量系统级性能指标
  - 结果不具备可比性

### 验证原则：编译成功 ≠ 验证成功

- **编译成功**：代码语法正确，类型检查通过
- **验证成功**：功能符合预期，性能达到要求
- **验证步骤**：
  1. 启动完整服务（Master + Volume + FUSE）
  2. 执行测试用例
  3. 验证结果正确性
  4. 记录测试日志和指标

## 连接管理原则

### 重连机制

- **所有 gRPC 调用必须有重试机制**：
  - 网络抖动时自动重试
  - 连接失效时自动重建
  - 设置合理的重试次数和间隔

- **gRPC keepalive 配置**：
  - 定期发送心跳检测连接状态
  - 配置合理的心跳间隔和超时时间
  - 及时清理失效连接

### 日志要求

- **连接事件必须记录**：
  - 连接建立成功/失败
  - 连接断开原因
  - 重连尝试和结果

- **请求事件必须记录**：
  - 请求参数（关键信息）
  - 请求耗时
  - 请求失败原因

## 持久化原则

### 数据一致性

- **写入必须持久化到磁盘**：
  - 元数据写入 Master 的 RocksDB
  - 数据写入 Volume 的磁盘文件
  - 确保服务重启后数据不丢失

- **缓存与持久化同步**：
  - 修改缓存后必须同步到持久化存储
  - 定期刷新脏数据
  - 确保缓存与持久化存储一致

## HPC Stripe 模式原则

### 条带化存储

- **数据条带化分布到多个 Volume**：
  - 提高并行读写性能
  - 均衡各个 Volume 的负载
  - 支持大文件的高效存储

- **条带配置**：
  - 条带大小：根据应用场景配置
  - Volume 数量：根据集群规模配置
  - 条带策略：顺序轮询或哈希分布
